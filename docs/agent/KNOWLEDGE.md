## Repo Shape

- `src/app/page.tsx` renders the homepage prediction, recommendation, and post-match review surfaces.
- `src/components/BankrollPlan.tsx` renders bankroll and market-specific buying guidance.
- `src/lib/model.ts` builds match predictions.
- `src/lib/model-iteration.ts` summarizes finished-match feedback and calibration signals.
- `src/lib/selection.ts` turns predictions into buying candidates.

## Model/Product Principles

- The project should help improve predictions over time, not just display static picks.
- Every finished-match surface should help answer: what did the model predict, what happened, why, and what should be improved next.
- Model confidence should stay honest. Separate a recommendation to try from a verified fix.
- Current user betting posture is single-match only: win/draw/loss direction, likely scorelines, confidence, and risk notes. Do not present multi-match, hedge-buy, or double-chance wording as recommendations.
- Missing or stale odds must be called out clearly; do not fabricate external prices, injuries, or lineups.
- Smart-money commitment is a derived signal, not a claimed handle/feed. Current implementation scores model-market agreement, multi-provider consensus, odds drift, edge, freshness, buying window, data quality, and draw avoidance. If actual opening/current odds movement is missing, say so in the card instead of inventing flow.
- The local odds importer preserves each provider's earliest observed current price as an `opening` baseline when no opening row exists. Drift should be described as local baseline/current movement unless an external true opener is explicitly imported.
- The homepage single-match board exposes this as a compact `盘口移动` badge. This is a visibility layer over the existing commitment signal, not a separate bet type.
- `盘口趋势` is calculated from same-provider baseline, previous current, and latest current odds. Sustained pressure can improve commitment; late pullback or sustained reverse movement should downgrade the buy signal.
- Scoreline probabilities are no longer raw independent Poisson only. The adjusted matrix intentionally reduces over-ranked clean sheets when the opponent has enough xG and lifts plausible BTTS paths, then derives top scores, BTTS, 1X2, and goal-range decisions from that same matrix.
- Score prediction includes a group-stage motivation heuristic. Build the group table only from completed results before the fixture, then boost attack for teams that still need points/goals to qualify and reduce tempo/add draw pressure for teams that are already safe. This is not a full qualification-scenario simulator; keep it honest unless a proper scenario engine is added later.
- Knockout score prediction is stage-aware and should be treated as 90-minute betting logic, not advancement probability. It lowers tempo, raises draw pressure, supports low-score draw/one-goal paths, and damps big clean-sheet scorelines. Keep advancement/champion probabilities separate on `/bracket`.
- `/groups` is now a group-stage review page, not the main forward-looking group simulator. It should retain final tables and every group match's actual result, prediction direction/probability, and top-three scorelines for later calibration.
- User-facing score prediction areas should show the top-three scorelines with probabilities via `ScorelineChips`, not a single `likelyScore` label. For future knockout matches, show the raw model top-three score probabilities and do not filter them to match the selected 1X2 direction. Group-stage review can keep the direction-filtered historical display where explicitly intended.
- Current trade surface uses three market plans per match: `双方进球`, `胜平负`, and `进球数范围`. These are separate single-match markets, not parlays.
- Trade actions should be non-money labels only: `主选`, `备选`, `观察`, `放弃`. Do not show `0.5u`, `1u`, or `等临场确认` on the user-facing board.
- BTTS signals near 52% should be labeled `观察`, not `不买`; the explanation should say the signal is only slightly above 50% and not strong enough for a primary selection.
- Daily reporting is available with `npm run export:trade-report`, writing `.local/trade-report.json`. It includes three-market active-signal success rates, probability-bin accuracy, market gaps, draw candidates, top-three scoreline accuracy, a blind-draw benchmark, and a conditional-draw benchmark.
- Treat the "buy every World Cup draw" idea as a benchmark to test, not as a rule. It is only profitable if actual draw probability exceeds the odds-implied probability after market margin; use the report's blind-draw ROI as evidence, and keep sample-size caveats visible.
- Pattern-fit is the second gate after commitment. It can downgrade a pick when this World Cup's learned pattern shows high draw risk, favorite overconfidence, elevated model temperature, favorite blank risk, or top-three scorelines that do not support the selected side.
- The homepage three-market cards should show only market label, selected side/range, and probability. Do not show buy/observe/main/backup action labels, stake sizes, or action-note text in those cards.
- After group stage, `src/data/generated-data.json` includes manual round-of-32 fixtures M073-M088. Prediction/report target windows use today+tomorrow Beijing dates, not tomorrow only, so current-day knockout matches stay visible. M073 South Africa 0-1 Canada is a supplemental result in `src/data/result-supplements.json`.
- Manual refresh must never dump command output into the UI. `/api/local-refresh` should return a short clean JSON status; detailed external-fetch failures belong in logs.
- `docs/model-reflections.md` is the lightweight reflection ledger. Update it when a model miss creates a durable rule worth checking before later predictions.
- `/review` rows include `reflectionDetail`; keep it concrete enough to explain what failed and what the next prediction should recheck.
- `npm run check:scorelines` checks `.local/trade-report.json` and fails if displayed top-three scorelines contradict the selected 1X2 side. Run it after scoreline/model changes.
- `npm run check:current-predictions` checks `.local/trade-report.json` and fails if any current prediction has already kicked off. Run it after prediction-window, homepage, matches, review, or trade-report changes.
- `npm run check:prediction-board` exports the report and runs both prediction-board guards. `npm run refresh:tomorrow` now runs those guards automatically after report export.
- `npm run check:site-health` checks the six key local pages for HTTP 200 and stale/error text. `npm run check:site` runs both prediction-board and page-health checks.
- Do not run `next build` while `next dev` is active. They share `.next` and can create transient missing chunk errors. Stop dev, clean `.next` if needed, build, then restart the local site.
- When openfootball lags knockout results, add only verified final scores to `src/data/result-supplements.json`; `npm run refresh:results` merges those supplements into the same automatic result path.
- `/bracket` has a knockout per-match prediction/result table like `/groups`; keep match, actual score, pre-match direction, direction probability, and raw model top-three scorelines visible.
- `/bracket` score history intentionally preserves M073-M084 with the original Baseline direction-filtered Top3 and uses Hybrid V2 raw Top3 from M085 onward. Score-hit statistics must follow that same per-match split and compare against 90-minute results.
- Kicked-off matches without locked results must be shown as waiting for results, not current predictions and not completed review rows. The report source of truth is `pendingResults`; homepage and `/review` should surface the same state.
- Polymarket/prediction-market odds are supported through CSV import with `marketKind=prediction_market` and probability fields, and through `POST /api/odds/polymarket/refresh`. The automatic refresh uses Polymarket Gamma markets and only imports when a local fixture has matched home/draw/away prices.
- Nansen prediction-market odds are supported through `POST /api/odds/nansen/refresh` with local `NANSEN_API_KEY`. The endpoint uses Nansen's market screener and only imports complete matched home/draw/away fixture rows as `prediction_market`; current verified result is 0 imported rows when Nansen has no complete local 1X2 match.
- Polymarket/Nansen World Cup winner markets are saved to `.local/team-market-strength.json` and used as a light long-term team-strength adjustment. Keep the multiplier small; winner markets are not single-match odds.
- Manually summarized chain smart-wallet probabilities are supported with `marketKind=smart_wallet`; automatic wallet clustering still requires a real chain analytics source/API key or a user-provided wallet list. Do not infer smart-wallet flow from generic market prices.
- Do not add reinforcement learning until the project has a much larger stable sample and a precise reward definition. Prefer explicit online calibration from completed matches, probability bins, and review rules for now.
- Baseline model identity is `baseline-v1-market-elo`. `market-only-v1` and `hybrid-v2-knockout` are local experimental comparison models only; do not replace Baseline until strict pre-match snapshots produce enough completed-match evidence.
- Pre-match snapshots live in local SQLite tables `prediction_snapshots` and `prediction_input_bundles`. Capture with `npm run snapshots:capture`; inspect with `npm run snapshots:stats`. Keep capture non-blocking and local-only unless public deployment is explicitly authorized.
- Use `npm run check:prediction-snapshots` after changing snapshot capture/storage. It verifies pre-kickoff timing, duplicate prevention, compact-vs-final matrix storage, input bundle presence, scoreline payload size, model versions, and lambda total/difference arithmetic.
- Model comparison cards must separate real inputs from absent inputs. If lineup, injury, weather, venue, over/under, handicap, or BTTS market data is not actually ingested, show it as missing/not included; never encode missing context as a numeric zero contribution.

## Local Site Startup

- `http://127.0.0.1:3000/` is a local server, not a static website. After Windows shuts down, the page is unavailable until the server starts again.
- `scripts/start-local-site.ps1` is the project-local launcher. It checks whether port 3000 is already listening and only starts `npm run dev -- --hostname 127.0.0.1 --port 3000` when needed.
- In the Codex sandbox, duplicate process-level `Path` and `PATH` can break PowerShell process spawning. The launcher intentionally removes duplicate `PATH` in-process before calling `Start-Process`.
- Do not add `-UseNewEnvironment` to the launcher; it caused Node CSPRNG startup failure on this machine.
- Windows scheduled task `WorldCupPredictorLocalSite` runs the launcher when user `LAPTOP-81RS0KNQ\leBron james` logs in. It does not require Codex to remain open.
- The scheduled task should keep two delayed logon triggers, one at `PT30S` and one at `PT2M`, with `StartWhenAvailable` enabled. This avoids the local site staying down when the first logon moment happens before the Windows/user environment is ready.
- The launcher waits up to 90 seconds for Next.js to bind port 3000. Do not shorten this; cold startup after reboot can be slower than an interactive restart.
- Startup logs are in `.local/local-site.log`; Next.js server logs are in `.local/dev-server.log` and `.local/dev-server.err.log`.
- There is also a Startup-folder fallback command at `%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\WorldCupPredictorLocalSite.cmd`; it runs the same `scripts/start-local-site.ps1` launcher if the scheduled task does not start after reboot.
- `scripts/refresh-results.mjs` now supports configured web result fallback through `src/data/result-web-sources.json`. Keep it conservative: only parse a result when both teams and a score are present; skip flaky pages instead of blocking refresh.
- `scripts/refresh-results.mjs` is the authoritative local result sync entry point. It first uses openfootball, then verified supplements, then configured web event fallbacks. It writes final scores into `overrides` and writes audit rows into `result_sync_status` with external match ID, status, score, source URL, last check, error, and post-match status.
- Each configured web-source check also writes a raw `result_sync_events` row. Use this table when diagnosing source delay, parse misses, or future source conflicts.
- The result sync has a `.local/result-sync.lock` process lock, retries, and 10-second request timeout. If one run is active, a second run exits cleanly instead of racing writes.
- Current web fallback examples use theScore event pages in `src/data/result-web-sources.json`: M080 England 2-1 DR Congo, M081 Belgium 3-2 Senegal, and M082 USA 2-0 Bosnia & Herzegovina. These exist because openfootball lagged these knockout results.
- Windows scheduled task `WorldCupPredictorResultSync` runs result sync every 15 minutes through `wscript.exe scripts\run-result-sync-hidden.vbs`. Do not switch it back to direct `powershell.exe`; even hidden PowerShell can flash a console window. Logs are `.local/result-sync.log` and `.local/result-sync.err.log`. `scripts/start-local-site.ps1` also runs one compensation result refresh before starting the local site, so reboot recovery updates results even before the first periodic task.
- Hosted deployments can trigger result sync through `POST /api/cron/result-sync`. Set `CRON_SECRET` and send either `Authorization: Bearer <secret>` or `x-cron-secret: <secret>` from the cron provider.
- Frontend waiting-result sections should only appear when a match has kicked off and no locked result exists. They should disappear after `overrides` gets the final score; `/review` should then show the completed match.
- `docs/group-stage-reflection.md` records group-stage prediction hit rates and miss patterns.
- The 15-minute Windows sync can run successfully while still failing to fetch a new finished match if the active source cannot identify that fixture. This was the old OpenFootball + manually configured theScore event-ID weakness, not a PowerShell failure. API-Football is now the intended primary source because it uses `fixture.id`; until the key has 2026 access, old fallback sources remain partial.
- `npm run sync:api-football` stores API usage in `.local/api-football-sync.json` with `requestDay`, `requestsUsedToday`, and `requestLimit=96`. If this guard trips, do not increase task frequency; fix source access or run one manual sync after the quota resets.
- New 2026-07-04 direction is local-only Cloudflare-compatible ESPN sync. Do not extend the Windows Task Scheduler path for new sync features. Build toward `workers/worldcup-sync/`, ESPN scoreboard ingestion, D1-compatible schema, local Wrangler-style handlers, and deploy guards. Public `workers.dev` must not be modified until the user explicitly approves deployment.
